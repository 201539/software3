# AI Coding Agent Web 应用 MVP 最小实现方案

## 1. 目标

本 MVP 的目标不是实现完整的 Agent Web 平台，而是验证最核心的一条闭环：

1. 用户输入需求
2. Agent 生成项目骨架
3. 用户看到工作区文件树和代码内容
4. 用户继续对话让 Agent 修改代码
5. Agent 通过受控工具完成文件修改和命令执行
6. 用户能看到结果和变更摘要

只要这条闭环跑通，就说明产品方向成立。

---

## 2. MVP 范围

### 2.1 必做功能

#### 2.1.1 Chat 输入与流式输出
- 用户可以输入自然语言需求
- Agent 可以流式返回思考结果、计划和执行结果
- 能展示任务状态

#### 2.1.2 工作区文件树
- 展示当前项目文件列表
- 支持点击打开文件
- 支持刷新工作区状态

#### 2.1.3 代码编辑器
- 支持查看文件内容
- 支持手动编辑并保存
- 支持展示 Agent 修改后的内容

#### 2.1.4 项目骨架生成
- Agent 能根据用户需求生成一个基础项目结构
- 至少支持一个清晰的默认模板，例如：
  - Next.js Web 项目
  - 或 Vite + React 项目
- 生成后能打开、查看、继续修改

#### 2.1.5 Agent 修改代码
- 用户继续对话提出修改需求
- Agent 读取相关文件
- Agent 生成 patch / diff
- Agent 应用修改
- Agent 返回修改摘要

#### 2.1.6 受控命令执行
- 支持运行必要的开发命令
- 例如：`npm install express`、`npm run build`、`python test.py`
- 命令必须在工作区内执行
- 命令执行结果要回显到界面

#### 2.1.7 版本快照与回滚
- 每轮重要修改后可创建快照
- 支持回滚到上一个稳定版本

---

## 3. MVP 不做什么

为了尽快验证核心价值，MVP 暂时不做：

- 多人协作
- 实时协同编辑
- 插件市场
- 复杂权限体系
- 企业组织管理
- 多模板复杂市场
- 过多模型切换
- 复杂计费系统
- 完整审计后台
- 高级终端功能

---

## 4. 最小架构

MVP 可以先做成一个轻量单体，但内部按 Agent 责任拆分。

### 4.1 必要模块

1. **Web UI**
   - Chat
   - 文件树
   - 编辑器
   - 状态面板
   - 日志面板

2. **Agent Orchestrator**
   - 任务规划
   - 工具调度
   - 状态流转
   - 汇总结果

3. **Context Builder**
   - 选择相关文件
   - 压缩长上下文
   - 给模型提供最小必要信息

4. **Tool Gateway**
   - 读文件
   - 写文件
   - patch 文件
   - 搜索文本
   - 运行命令
   - 创建快照
   - 回滚快照

5. **Workspace Manager**
   - 管理项目目录
   - 保存文件
   - 记录版本
   - 维护快照

6. **Preview Runner**
   - 启动项目预览
   - 返回预览 URL
   - 收集构建错误

7. **State Store**
   - 保存项目、会话、任务、消息、版本、日志

## 5. MVP 技术选型

### 5.1 Web UI 技术栈

我已选定：
- `React` + `TypeScript`
- `Monaco Editor`
- `WebSocket` 做实时消息和任务状态同步

### 5.2 Agent Runtime 层实现语言

我已选定：
- `Node.js` + `TypeScript`

### 5.3 工作区执行方式

我已选定：
- 本地工作区 + 本地 shell
- 受控工作区目录内执行命令
- 通过工具层封装文件和命令操作

### 5.4 存储方式

我已选定：
- 先轻量存储
- 本地文件
- JSON
- 简单数据库

说明：MVP 阶段优先保证可用和可回滚，不一开始强依赖重型存储方案。

### 5.5 任务编排方式

我已选定：
- 任务编排借助现成 Agent 框架

说明：
- 重点不是从零造轮子
- 而是尽快验证“规划 -> 工具调用 -> 修改 -> 验证 -> 总结”的闭环
- 现成框架只用于提高编排效率，核心工作区能力和工具安全仍由我们自己掌控

### 5.6 编辑器

我已选定：
- `Monaco Editor`

### 5.7 其他技术原则

- 先不强依赖复杂 RAG
- 上下文先用启发式选择
- 命令执行采用风险分级
- 版本快照与回滚必须保留

## 6. 目录结构设计

### 6.1 设计原则

目录结构要满足以下要求：

- 适合 `React + TypeScript + Node.js + TypeScript`
- 方便 Agent 代码生成和后续维护
- 让 Web UI、Agent Runtime、工具层、工作区逻辑分开
- 保持 MVP 轻量，不要一开始拆得过碎
- 目录名尽量直观，减少 Cursor 理解成本

### 6.2 推荐仓库结构

```text
repo/
├─ apps/
│  ├─ web/                      # React + TypeScript 前端
│  └─ runtime/                  # Node.js + TypeScript Agent Runtime
│
├─ packages/
│  ├─ shared/                   # 前后端共享类型、常量、工具
│  ├─ agent-core/               # Agent 编排、Planner、Executor、Reviewer
│  ├─ tool-gateway/             # 文件、命令、快照等工具接口
│  ├─ workspace-manager/        # 工作区管理、版本管理、文件索引
│  └─ context-builder/          # 上下文选择与压缩
│
├─ workspaces/                  # 本地项目工作区根目录
│  └─ {projectId}/
│     ├─ workspace/             # 当前工作区代码
│     ├─ snapshots/             # 历史快照
│     ├─ logs/                  # 命令和 Agent 日志
│     └─ versions.json          # 版本索引
│
├─ data/                        # 轻量存储区域
│  ├─ app.json                  # 应用级配置
│  ├─ projects.json             # 项目索引（可选）
│  └─ sessions.json             # 会话索引（可选）
│
├─ scripts/                     # 初始化、导入、清理、辅助脚本
├─ docs/                        # 方案、设计、说明文档
├─ .env.example                 # 环境变量示例
├─ package.json                 # Monorepo 根配置
├─ tsconfig.base.json           # TypeScript 基础配置
└─ README.md
```

### 6.3 各目录职责说明

#### `apps/web`
前端应用，负责：
- Chat
- 文件树
- 编辑器
- 任务状态
- 日志展示
- 预览信息展示

#### `apps/runtime`
Agent 运行时，负责：
- 接收用户请求
- 调度 Agent 流程
- 调用工具层
- 管理工作区
- 返回实时结果

#### `packages/agent-core`
Agent 核心逻辑，负责：
- Planner
- Executor
- Reviewer
- Summarizer
- 任务状态机

#### `packages/tool-gateway`
统一封装可执行工具，负责：
- 读写文件
- patch 文件
- 搜索内容
- 运行命令
- 创建快照
- 回滚快照

#### `packages/workspace-manager`
负责工作区的生命周期管理：
- 创建项目目录
- 初始化模板
- 生成文件索引
- 保存快照
- 读取版本信息

#### `packages/context-builder`
负责：
- 文件相关性选择
- 长文本压缩
- 上下文预算控制
- 生成本轮模型输入包

#### `workspaces/`
这是所有真实项目代码的落点。每个项目独立目录，避免互相干扰。

#### `data/`
用于放轻量级 JSON 或应用配置。MVP 阶段只放必要内容，不把这里做成完整数据库替代品。

### 6.4 单项目内部结构

每个项目的工作区建议长这样：

```text
workspaces/{projectId}/
├─ workspace/
│  ├─ package.json
│  ├─ src/
│  ├─ public/
│  └─ ...
├─ snapshots/
│  ├─ v1/
│  ├─ v2/
│  └─ ...
├─ logs/
│  ├─ agent.log
│  └─ command.log
└─ versions.json
```

### 6.5 版本索引文件建议

`versions.json` 负责记录版本元数据，不保存完整代码内容。

示例：

```json
[
  {
    "id": "v1",
    "name": "initial scaffold",
    "description": "初始项目骨架",
    "snapshotPath": "snapshots/v1",
    "createdAt": "2026-04-20T10:00:00Z",
    "taskId": "task_001"
  },
  {
    "id": "v2",
    "name": "add login page",
    "description": "增加登录页和路由",
    "snapshotPath": "snapshots/v2",
    "createdAt": "2026-04-20T10:30:00Z",
    "taskId": "task_002"
  }
]
```

### 6.6 为什么这个结构适合 MVP

- 足够清晰，便于 Cursor 辅助开发
- 前端、Agent、工具、工作区边界明确
- 不会一开始就过度拆分
- 后续如果要拆服务也容易迁移
- 真实代码资产和轻量状态文件分离，利于回滚和调试

## 7. 工具接口设计

MVP 的工具层不需要很多接口，但需要把每个接口的职责、输入、输出和边界定义清楚，避免 Agent 在执行时随意拼命令或绕过工作区。

### 7.1 设计原则

- 工具职责单一
- 只允许操作当前项目工作区
- 输入输出尽量结构化
- 所有工具调用都要可审计
- 命令执行必须经过风险控制
- 工具返回值要足够让 Agent 继续决策

### 7.2 基础工具接口

#### 7.2.1 `list_files`
用途：列出工作区某个目录下的文件和子目录。

输入：
- `projectId`
- `path`

输出：
- 文件/目录列表
- 每项的类型、路径、大小、修改时间

边界：
- 只能读取当前项目工作区内的路径
- 不能越权访问其他目录

#### 7.2.2 `read_file`
用途：读取指定文件内容。

输入：
- `projectId`
- `path`

输出：
- 文件文本内容
- 文件编码信息（可选）
- 文件大小

边界：
- 只能读取工作区内文件
- 大文件可返回摘要或分片结果

#### 7.2.3 `write_file`
用途：将完整内容写入指定文件。

输入：
- `projectId`
- `path`
- `content`

输出：
- 写入是否成功
- 变更后的文件哈希

边界：
- 仅限工作区内路径
- 写入前应记录旧内容或创建快照

#### 7.2.4 `patch_file`
用途：根据 diff 或局部补丁修改文件。

输入：
- `projectId`
- `path`
- `patch`

输出：
- 是否应用成功
- 应用前后摘要
- 变更行数

边界：
- 适合局部修改
- 如果 patch 失败，应返回失败原因而不是静默忽略

#### 7.2.5 `search_in_workspace`
用途：在工作区内搜索文本或关键字。

输入：
- `projectId`
- `query`
- `path`（可选）

输出：
- 命中结果列表
- 命中文件路径
- 命中行号或片段

边界：
- 只搜索当前工作区
- 结果要限制数量，避免过长

#### 7.2.6 `run_command`
用途：在受控工作区中执行开发命令。

输入：
- `projectId`
- `command`
- `cwd`（默认工作区根目录）
- `timeoutMs`（可选）

输出：
- `exitCode`
- `stdout`
- `stderr`
- `durationMs`
- `riskLevel`
- 是否需要用户确认

边界：
- 只能在工作区内执行
- 必须经过命令结构化校验
- 必须做风险分级
- 高风险命令要拦截或确认

#### 7.2.7 `create_snapshot`
用途：创建工作区快照。

输入：
- `projectId`
- `name`（可选）
- `description`（可选）

输出：
- `snapshotId`
- `snapshotPath`
- 创建时间

边界：
- 快照必须对应当前工作区状态
- 创建快照前应保证当前修改已落盘

#### 7.2.8 `restore_snapshot`
用途：从指定快照恢复工作区。

输入：
- `projectId`
- `snapshotId`

输出：
- 是否恢复成功
- 恢复后的工作区状态

边界：
- 恢复前应提示或确认
- 恢复操作应可审计

### 7.3 工具返回结构建议

为了让 Agent 更容易处理，工具返回值建议统一包含：
- `success`
- `data`
- `error`
- `meta`

示例：
- `success`: 是否成功
- `data`: 业务数据
- `error`: 错误信息和错误码
- `meta`: 耗时、风险等级、路径等附加信息

### 7.4 最小实现优先级

MVP 阶段建议优先实现：
1. `list_files`
2. `read_file`
3. `write_file`
4. `patch_file`
5. `run_command`
6. `create_snapshot`
7. `restore_snapshot`
8. `search_in_workspace`

### 7.5 为什么要先定义工具接口

因为 Agent 能不能真正干活，取决于它能不能稳定调用工具。
工具接口越清楚，后面实现规划、执行、验证、回滚就越稳。

## 8. Agent 状态机设计

MVP 里的 Agent 不应该是“随便聊一下然后顺手改代码”，而应该是一个**状态明确、流程可控、可恢复**的任务执行系统。

### 8.1 状态设计原则

- 状态要少，但要够用
- 每个状态都要有明确职责
- 状态流转必须由 Orchestrator 控制
- 出错后可以回退或重试
- 前端可以基于状态展示进度

### 8.2 建议状态

#### `idle`
当前没有正在执行的任务。

#### `planning`
Agent 正在分析用户需求并生成计划。

#### `context_loading`
正在读取文件树、相关文件、历史变更、日志等上下文。

#### `editing`
正在生成或应用文件修改。

#### `validating`
正在运行命令、检查结果或预览构建。

#### `waiting_user`
当前需要用户确认、补充信息或决定是否继续。

#### `succeeded`
任务已成功完成。

#### `failed`
任务执行失败，等待重试、修复或回滚。

#### `rolling_back`
正在恢复快照或回滚修改。

#### `cancelled`
任务被用户取消或系统终止。

### 8.3 状态流转图

```text
idle
  ↓
planning
  ↓
context_loading
  ↓
editing
  ↓
validating
  ├─ 成功 → succeeded → idle
  ├─ 需要确认 → waiting_user → planning / editing
  ├─ 失败可修复 → editing / planning
  └─ 需要回滚 → rolling_back → idle
```

### 8.4 各状态职责

#### `planning`
- 解析用户意图
- 生成结构化步骤
- 决定本轮要做什么

#### `context_loading`
- 读取项目文件树
- 读取相关文件
- 读取最近 diff 和日志
- 压缩上下文

#### `editing`
- 生成 patch
- 写入文件
- 应用变更
- 创建快照

#### `validating`
- 运行命令
- 读取 stderr/stdout
- 判断是否通过

#### `waiting_user`
- 提示用户确认高风险操作
- 要求补充信息
- 让用户选择继续或回滚

#### `succeeded`
- 输出总结
- 记录版本
- 回到空闲状态

#### `failed`
- 记录失败原因
- 保留日志和上下文
- 允许重试或回滚

#### `rolling_back`
- 恢复快照
- 回写版本索引
- 清理当前失败状态

### 8.5 状态机的最小实现要求

- 每个任务必须有当前状态
- 每次状态变化都要记录时间和原因
- 前端能看到任务现在处于哪个阶段
- 失败时必须保留可追踪信息
- 状态机必须可恢复，不能只存在内存里

### 8.6 状态机与任务模型的关系

一个 `AgentTask` 至少应包含：
- 任务 id
- 项目 id
- 当前状态
- 输入需求
- 计划摘要
- 相关文件
- 工具调用列表
- 失败原因
- 版本快照 id
- 开始时间和结束时间

### 8.7 为什么状态机重要

状态机的作用是把 Agent 的行为从“不可控聊天”变成“可跟踪任务”。

有了状态机，你才能：
- 看懂任务进度
- 做到失败重试
- 支持回滚
- 支持前端展示
- 支持后续扩展更复杂的 Agent 流程

## 9. 最小用户流程

### 9.1 新建项目

1. 用户输入需求
2. Agent 生成计划
3. Agent 创建项目工作区
4. Agent 写入项目骨架
5. UI 刷新文件树

### 9.2 继续修改

1. 用户继续输入修改需求
2. Agent 读取相关文件
3. Agent 生成 patch
4. Agent 应用修改
5. 需要时运行命令验证
6. UI 展示变更摘要

### 9.3 查看结果

1. 用户查看文件树
2. 用户打开文件
3. 用户查看日志和 diff
4. 用户决定继续修改或回滚

## 10. Agent 最小能力实现

### 10.1 规划能力

MVP 里只需要一个简单 Planner：
- 输入用户需求
- 输出结构化步骤
- 只规划本轮要做的事情

### 10.2 上下文能力

MVP 里采用启发式上下文选择：
- 当前文件优先
- 最近修改文件优先
- 报错命中文件优先
- 同模块文件优先

先不强依赖复杂 RAG。

### 10.3 修改能力

MVP 里优先采用 patch 方式：
- 先读文件
- 再定位
- 再生成 diff
- 再写回

### 10.4 总结能力

每轮执行结束后输出：
- 改了什么
- 改了哪些文件
- 是否通过验证
- 下一步建议

## 11. 命令执行的最小安全方案

MVP 不要做成“任意 shell 全开”，也不要做成死板白名单。

### 11.1 最小可行策略

- 只允许在工作区内执行
- 只允许 Agent 通过工具调用执行
- 对命令做结构化解析
- 按风险分级
- 高风险命令要求确认
- 所有执行记录日志

### 11.2 MVP 支持的命令

建议先支持这些常见命令模板：
- `npm install <pkg>`
- `pnpm add <pkg>`
- `npm run build`
- `npm run lint`
- `python <file>`
- `pytest -q`

### 11.3 MVP 明确禁止的行为

- 工作区外路径访问
- 任意系统命令执行
- 通过拼接用户原文直接执行 shell
- 删除大量文件
- 修改系统配置

## 12. 数据模型最小集合

### 12.1 必要对象

- `User`
- `Project`
- `Conversation`
- `Message`
- `AgentTask`
- `ToolCall`
- `VersionSnapshot`
- `FileIndex`

### 12.2 MVP 只保存必要信息

- 项目基本信息
- 当前工作区路径
- 对话记录
- 每轮任务状态
- 工具调用日志
- 快照位置和摘要
- `versions.json` 中的版本索引信息

### 12.3 MVP 版本管理方案

MVP 阶段只采用：
- **文件快照**：每轮重要修改后保存一份工作区快照
- **简单 JSON 版本索引**：用一个 `versions.json` 记录版本 id、名称、时间、说明、快照路径

不在 MVP 阶段引入完整数据库版本表。

## 15. 是否现在直接实现

### 15.1 我的建议

我建议 **先做最小实现，不要继续大幅细化设计**。

原因是：
- 你现在已经把关键技术路线定下来了
- MVP 的闭环和边界已经足够清楚
- 再细化很容易进入“设计过度”
- 真正不确定的点，只有实现后才会暴露

### 15.2 适合现在就开始实现的前提

你已经明确了：
- Web UI 技术栈
- Agent Runtime 语言
- 工作区执行方式
- 存储方式
- 实时通信方式
- 编辑器
- 任务编排方式
- 版本管理方式

这意味着可以进入实现阶段。

### 15.3 仍建议在实现前补一小段的内容

不需要再写大文档，但建议在编码前再补 3 个小设计：
1. `versions.json` 结构
2. Agent 任务状态机
3. Tool Gateway 的命令输入输出格式

这三项属于“实现前的最后一层细化”，不会拖慢进度，但能显著减少返工。

## 16. MVP 成功标准

MVP 成功，不是看功能有多少，而是看下面这条链路是否顺畅：

- 输入需求
- 自动生成项目骨架
- 自动展示文件结构
- 可继续对话修改
- 可以运行命令验证
- 可以看到结果和摘要
- 可以回滚

如果这条链路成立，就可以进入下一阶段优化。

---

## 17. MVP 风险点

### 17.1 最大风险

- Agent 不能稳定改代码
- 命令执行不安全
- 上下文选错导致乱改
- 快照和回滚不可靠
- 预览环境不稳定

### 17.2 对应策略

- 先做小闭环，不做复杂功能
- 命令执行分级管控
- 上下文优先启发式，必要时再加检索
- 所有修改都可回滚
- 预览先支持一个模板项目即可

---

## 18. 最终结论

这个 MVP 的核心不是“功能最全”，而是“最小闭环跑通”。

只要能实现：

**用户输入需求 -> Agent 生成项目骨架 -> 用户继续对话修改 -> Agent 受控执行命令和文件变更 -> 用户看到结果并可回滚**

就说明产品方向可行，可以继续迭代到更完整的 Agent Web 应用。
