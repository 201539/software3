# AI Coding Web 软件从 0 到 1 流程

本文档记录的是：如果你要做一个 AI Coding Web 软件，并且在整个过程中会大量使用 Cursor 辅助编码，应该怎么从 0 开始推进、先做什么、后做什么、每一步要产出什么、如何避免返工。

---

## 1. 先明确你要做的产品是什么

在写代码之前，先把目标说清楚。否则很容易做到一半发现方向不对。

### 1.1 先回答这 6 个问题

1. 你的产品主要服务谁？
   - 独立开发者
   - 前端工程师
   - 全栈工程师
   - 团队协作场景
   - 初学者

2. 你希望它解决什么问题？
   - 帮用户更快写代码
   - 帮用户理解代码
   - 帮用户生成项目结构
   - 帮用户修改现有项目
   - 帮用户做代码审查
   - 帮用户自动补全业务功能

3. 你打算做成什么形态？
   - Chat + 代码编辑器
   - Chat + 项目面板
   - 代码仓库同步 + AI 改码
   - 类似 Cursor 的在线 IDE
   - 只做某个垂直场景，如“生成后台管理系统”

4. 你的核心差异化是什么？
   - 比 Cursor 更轻量
   - 比 Cursor 更适合中文用户
   - 比 Cursor 更适合某类项目
   - 比 Cursor 更强调协作
   - 比 Cursor 更适合非技术用户

5. 你的 MVP 是什么？
   - 最小可用版本只做一件事
   - 例如：用户输入需求，系统生成项目骨架并可继续对话修改

6. 你未来怎么赚钱？
   - 订阅制
   - 按调用次数计费
   - 企业版
   - 私有部署
   - 模板市场 / 插件市场

### 1.2 这一阶段的输出物

你最终应该写出一页纸的产品定义，包括：
- 产品名称
- 目标用户
- 核心场景
- 核心价值
- MVP 功能范围
- 非目标范围
- 收费模式假设

## 2. 先做竞品和参考调研

不要急着写代码，先看市场上已经有哪些方案。

### 2.1 你要看的对象

建议至少看这些类型：
- Cursor
- GitHub Copilot
- Windsurf / Codeium
- Replit
- Claude Code / 类终端编码助手
- 各种 AI 建站 / AI IDE 产品

### 2.2 重点观察什么

每个产品都记录以下内容：
- 它的核心用户是谁
- 它解决的第一痛点是什么
- 首次打开的体验如何
- 它的主流程是什么
- 它的定价方式
- 它的留存机制是什么
- 它哪里让人觉得爽，哪里让人觉得烦

### 2.3 你要输出的东西

建议整理成一个对比表：
- 产品名
- 定位
- 核心功能
- 优点
- 缺点
- 值得借鉴的点
- 你要避开的点

### 2.4 这一步的目的

不是抄，而是找到：
- 你要切入的细分场景
- 你不做什么
- 你比别人更简单、更快、更懂某类用户的地方

---

## 3. 定义 MVP，不要一开始做大而全

AI coding web 软件最容易犯的错误，就是想一次性做成“在线 Cursor + GitHub + Copilot + Notion + 设计器”。

### 3.1 MVP 推荐原则

只保留一条主链路：
1. 用户进入网站
2. 输入目标或导入项目
3. AI 帮忙生成或修改代码
4. 用户查看结果
5. 用户继续迭代
6. 用户保存/导出/部署

### 3.2 MVP 里建议先有的功能

最基础可考虑：
- 登录/注册
- 创建项目
- 输入需求的对话框
- 代码生成
- 文件树展示
- 代码编辑器
- 版本记录
- 运行预览
- 错误提示
- 保存结果

### 3.3 MVP 里先不要做的功能

建议第一版先别做：
- 复杂协作
- 多人实时编辑
- 插件市场
- 太多模型切换
- 高级权限系统
- 企业组织架构
- 过于复杂的模板系统
- 过早做移动端适配

---

## 4. 先写产品文档，再写代码

这一步非常重要。Cursor 很适合写代码，但如果没有清晰文档，它也会把你带偏。

### 4.1 你至少要写 5 份文档

#### 4.1.1 PRD
写清楚产品要解决什么问题。

#### 4.1.2 用户流程
从用户进入网站到完成目标，完整画出路径。

#### 4.1.3 页面清单
列出需要哪些页面和页面之间怎么跳转。

#### 4.1.4 数据结构
至少想清楚：
- 用户
- 项目
- 会话
- 文件
- 版本
- 消费记录

#### 4.1.5 技术方案
包括前端、后端、数据库、模型接入、文件系统、部署方式。

### 4.2 文档写法建议

每份文档都尽量包含：
- 目的
- 范围
- 输入
- 输出
- 异常情况
- 优先级

---

## 5. 先设计整体架构

AI coding web 软件通常不是单纯前端页面，而是一个系统。

### 5.1 建议的系统模块

- 前端 Web 应用
- 后端 API 服务
- 认证系统
- 项目管理系统
- AI 编排层
- 代码存储层
- 版本管理层
- 任务队列 / 异步执行
- 预览/沙箱运行环境
- 日志和监控
- 支付系统

### 5.2 常见架构思路

#### 方案 A：最小单体架构
适合 MVP。
- 前后端分离，但后端先做成单体
- 数据库简单
- AI 调用直接由后端编排
- 预览功能先用基础容器或轻量沙箱

#### 方案 B：模块化架构
适合后续扩展。
- API 层
- AI Orchestrator
- 代码执行服务
- 文件服务
- 计费服务

### 5.3 推荐路线

如果你是从 0 开始，建议：
- 先做单体
- 接着模块化拆分
- 不要一开始就微服务化

## 6. 选技术栈

技术栈的原则不是“最潮”，而是“最稳、最好维护、最适合你当前能力”。

### 6.1 前端建议

常见选择：
- React / Next.js
- TypeScript
- Tailwind CSS
- Zustand / Redux Toolkit
- Monaco Editor
- TanStack Query

### 6.2 后端建议

常见选择：
- Node.js / NestJS / Express
- Python / FastAPI
- PostgreSQL
- Redis
- BullMQ / Celery

### 6.3 AI 接入建议

- 先统一抽象一个模型服务层
- 不要把模型调用散落在各处
- 设计好 prompt 模板、上下文拼接、错误重试
- 记录 token、耗时、成功率、失败原因

### 6.4 运行环境建议

如果你的产品需要运行用户代码，必须尽早考虑：
- 沙箱隔离
- 资源限制
- 超时控制
- 网络访问限制
- 文件系统隔离
- 安全审计

---

## 7. 先设计用户主流程

一个 AI coding web 软件，真正的核心不是页面数量，而是用户主路径是否顺畅。

### 7.1 推荐主流程

1. 用户注册/登录
2. 创建项目
3. 选择项目类型或输入需求
4. AI 生成项目初始化内容
5. 用户查看代码结构
6. 用户提出修改需求
7. AI 修改代码
8. 用户预览效果
9. 用户修复问题或继续迭代
10. 用户导出、部署或分享

### 7.2 每一步都要考虑的问题

- 用户现在想做什么？
- 这一步有没有必要？
- 用户下一步最可能做什么？
- 失败了怎么提示？
- 是否能让用户少点一次按钮？

---

## 8. 用 Cursor 的正确方式

Cursor 应该像“加速器”，不是“替你思考的唯一大脑”。

### 8.1 正确使用方式

#### 8.1.1 先让 Cursor 帮你拆任务
比如让它帮你：
- 梳理模块
- 列出文件结构
- 拆分开发顺序
- 生成接口草案

#### 8.1.2 再让 Cursor 写局部代码
例如：
- 某个页面组件
- 某个 API
- 某个工具函数
- 某个表单逻辑

#### 8.1.3 每次只改一个小目标
不要一次性让它重构整个项目。

### 8.2 你给 Cursor 的提示要具体

好的提示应该包含：
- 目标
- 输入
- 输出
- 限制
- 目录位置
- 现有代码上下文
- 不要做什么

### 8.3 推荐的工作方式

- 先出方案
- 再拆任务
- 再小步实现
- 每次改动后检查
- 出问题立刻回滚或修正

## 9. 从 0 开始的开发顺序

下面是推荐的实际执行顺序。

### 第 1 阶段：项目准备

#### 任务 1：确定产品方向
产出：一页纸产品定义。

#### 任务 2：写 PRD
产出：完整 PRD。

#### 任务 3：列页面和流程
产出：页面地图、用户流程图。

#### 任务 4：定技术栈
产出：技术选型清单。

#### 任务 5：建仓库
产出：初始化项目结构。

### 第 2 阶段：搭基础框架

#### 任务 6：搭建前端骨架
产出：基础布局、导航、首页、登录页、项目页。

#### 任务 7：搭建后端骨架
产出：健康检查接口、认证接口、项目接口。

#### 任务 8：接入数据库
产出：用户表、项目表、会话表、文件表。

#### 任务 9：接入鉴权
产出：登录、注册、会话管理。

### 第 3 阶段：做核心闭环

#### 任务 10：创建项目
产出：用户可创建新项目。

#### 任务 11：输入需求
产出：对话框或任务面板。

#### 任务 12：调用 AI 生成代码
产出：生成文件树和初始代码。

#### 任务 13：展示代码
产出：文件树 + 编辑器。

#### 任务 14：支持修改
产出：用户再提需求，AI 修改代码。

#### 任务 15：保存版本
产出：可回溯历史版本。

### 第 4 阶段：做预览与调试

#### 任务 16：添加运行预览
产出：可看页面效果。

#### 任务 17：处理错误
产出：运行报错提示和修复建议。

#### 任务 18：提升交互体验
产出：加载状态、进度提示、失败重试。

### 第 5 阶段：打磨和上线

#### 任务 19：做埋点
产出：知道用户在做什么、卡在哪里。

#### 任务 20：做计费
产出：免费额度、套餐、余额。

#### 任务 21：做部署
产出：测试环境、生产环境、CI/CD。

#### 任务 22：做监控
产出：错误监控、日志、告警。

#### 任务 23：正式发布
产出：公开上线版本。

## 10. 每个功能都要按这个模板开发

开发一个功能时，不要直接让 Cursor 开写，先走这个模板。

### 10.1 功能开发模板

1. 先写清楚功能目标
2. 列出用户故事
3. 定义输入输出
4. 定义边界情况
5. 设计接口
6. 设计 UI
7. 拆分文件
8. 编写实现
9. 自测
10. 修复问题
11. 再联调
12. 最后上线

### 10.2 示例

例如你要做“AI 修改代码”功能：
- 目标：根据用户描述修改指定文件
- 输入：需求文本、当前文件内容、上下文
- 输出：修改后的文件内容、变更摘要
- 边界情况：文件冲突、上下文过长、模型失败
- 接口：提交修改请求、返回 diff、保存版本

## 11. 使用 Cursor 时的高效工作法

### 11.1 推荐节奏

- 先想清楚再问 Cursor
- 一次只让它解决一个问题
- 让它先给方案，再写代码
- 写完后立刻检查
- 遇到偏题，及时纠正

### 11.2 最适合交给 Cursor 的任务

- 生成样板代码
- 写重复性组件
- 写接口层代码
- 写类型定义
- 写 CRUD
- 写测试
- 做重构
- 修小 bug

### 11.3 不要全交给 Cursor 的任务

- 产品方向判断
- 核心用户体验决策
- 定价策略
- 商业模式
- 最终架构取舍
- 安全策略

## 12. 你应该如何组织项目文件

建议一开始就保持清晰结构。

### 12.1 前端目录建议

- `app/` 或 `src/pages/`
- `components/`
- `features/`
- `hooks/`
- `lib/`
- `store/`
- `types/`
- `styles/`

### 12.2 后端目录建议

- `src/routes/`
- `src/services/`
- `src/controllers/`
- `src/models/`
- `src/repositories/`
- `src/utils/`
- `src/config/`

### 12.3 按功能分层

尽量按业务功能组织，而不是把所有文件都塞在一起。

## 13. 数据和状态要提前规划

AI coding 产品非常依赖状态管理。

### 13.1 你需要跟踪的数据

- 用户信息
- 项目元数据
- 当前会话
- 文件树
- 编辑历史
- AI 请求记录
- 版本记录
- 消费记录
- 预览状态

### 13.2 你需要考虑的状态问题

- 用户刷新页面后状态是否还在
- 多次修改如何合并
- AI 返回结果和本地编辑冲突如何处理
- 失败后是否可以重试
- 版本能不能回滚

---

## 14. 必须尽早做的安全和稳定性工作

如果你做的是 AI coding web 软件，这部分不能拖到最后。

### 14.1 安全问题

- 用户代码隔离
- Prompt 注入防护
- 文件访问边界控制
- API Key 安全存储
- 敏感信息脱敏
- 权限控制

### 14.2 稳定性问题

- 超时处理
- 重试机制
- 幂等性
- 限流
- 降级
- 失败日志

### 14.3 成本问题

- 模型调用成本
- 预览环境成本
- 存储成本
- 带宽成本
- 队列任务成本

---

## 15. 你应该怎么测试

不要等到最后才测。

### 15.1 测试层次

#### 单元测试
测函数、工具、业务逻辑。

#### 集成测试
测接口、数据库、AI 调用链路。

#### 端到端测试
测用户从登录到生成项目的完整流程。

#### 人工测试
真实走一遍主流程，找体验问题。

### 15.2 测试重点

- 登录流程
- 创建项目
- AI 生成
- AI 修改
- 保存版本
- 预览运行
- 错误恢复

---

## 16. 上线前你要检查什么

### 16.1 产品层

- 主流程是否完整
- 是否真的解决了问题
- 新用户是否能看懂
- 是否存在过多步骤

### 16.2 技术层

- 环境变量是否完整
- 日志是否可追踪
- 错误是否可观测
- 数据库迁移是否安全
- 备份是否可恢复

### 16.3 运营层

- 是否有引导页
- 是否有欢迎邮件
- 是否有埋点
- 是否有反馈入口
- 是否有客服或联系方式

---

## 17. 发布后的迭代顺序

第一版上线后，不要立刻扩功能，先观察。

### 17.1 先看这些数据

- 注册转化率
- 创建项目率
- AI 调用成功率
- 编辑后保存率
- 预览使用率
- 次日留存
- 付费转化率

### 17.2 先优化什么

优先级通常是：
1. 主流程卡点
2. AI 生成质量
3. 预览和错误修复体验
4. 保存和版本管理
5. 收费体系
6. 高级功能

---

## 18. 推荐你实际执行的“从 0 到 1”顺序

下面是最实用的简化版。

### 第一步
明确产品定位和目标用户。

### 第二步
写 PRD 和用户流程。

### 第三步
确定 MVP，只保留一条主链路。

### 第四步
选技术栈，定架构。

### 第五步
搭项目骨架和基础页面。

### 第六步
接入登录、数据库、项目管理。

### 第七步
做 AI 生成和 AI 修改闭环。

### 第八步
做预览、版本、错误处理。

### 第九步
做测试、埋点、监控。

### 第十步
上线，收反馈，继续迭代。

---

## 19. 给你的实战建议

1. 不要一开始就追求和 Cursor 一模一样。
2. 先做一个细分场景，做深做透。
3. 文档先行，代码后行。
4. 让 Cursor 干重复劳动，你负责方向和判断。
5. 每一步都要有产出物，不要空想。
6. 主流程一定要短，短，再短。
7. 先跑通，再优化。
8. 先验证需求，再扩展功能。
9. 每次只改一小块，降低返工。
10. 一定要尽早做日志、监控、版本回滚。

## 21. 最后一句话
