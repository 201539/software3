# MyClaw 后端流程说明（迭代二）

## 1. 文档目的

本文档用于说明在当前 `step2/myclaw.md` 设计方案下，后端团队需要做什么，以及后端在整个系统中的端到端流程。

目标：
- 明确后端职责边界；
- 明确请求从进入到产出结果的完整链路；
- 明确每个阶段的输入、处理、输出。

---

## 2. 后端总体职责

后端在 MyClaw 中承担 6 类核心职责：

1. **任务接入与管理**
   - 接收前端创建的任务；
   - 管理任务生命周期（创建、执行、重试、终止、归档）。

2. **Agent 编排与状态机控制**
   - 将任务转为可执行步骤；
   - 控制 `Plan -> Act -> Observe -> Reflect -> Re-plan -> Deliver` 状态流转。

3. **工具调用统一适配**
   - 统一封装外部工具调用（如网页操作、CI日志读取、Issue系统对接）；
   - 统一工具调用协议与错误处理规范。

4. **异常处理与自我修正执行**
   - 对超时、参数错误、目标不可达等异常进行分类；
   - 触发重试、降级、反思重规划。

5. **结果聚合与可追溯输出**
   - 汇总执行结论、关键证据、失败原因、建议动作；
   - 保存结构化 trace 供可视化展示与评估平台使用。

6. **数据持久化与查询服务**
   - 保存任务、运行记录、步骤轨迹、工具日志；
   - 提供前端查询接口和评估平台读取接口。

---

## 3. 后端分层架构（建议）

1. **API 层**
   - 提供任务创建、启动执行、状态查询、结果查询接口；
   - 做参数校验、鉴权、限流（可选）。

2. **Orchestrator（编排层）**
   - 负责任务状态机推进；
   - 调用推理模块与工具模块；
   - 控制重试与反思逻辑。

3. **Reasoning（推理层）**
   - 实现 Plan-and-Execute / ReAct / Reflexion；
   - 输出下一步动作与策略。

4. **Tool Adapter（工具层）**
   - 统一封装具体工具（例如 Playwright Adapter、CI Log Adapter、Issue Adapter）；
   - 统一返回格式：`success/data/error/latency`。

5. **Worker（异步执行层，可选）**
   - 执行长任务与重试队列；
   - 保障 API 快速响应与任务后台运行。

6. **Data（数据层）**
   - 任务表、运行表、步骤轨迹表、工具日志表；
   - 支撑结果回放和评估。

---

## 4. 端到端流程（核心）

## 4.1 任务创建阶段

**输入**：
- 任务类型（如 CI 失败巡检 / 回归验证 / 冒烟检查）
- 目标 URL 或目标对象
- 可选约束（超时、重试上限、优先级）

**后端处理**：
1. API 接收请求并做参数校验；
2. 创建 `task` 记录，状态置为 `pending`；
3. 返回 `task_id` 给前端。

**输出**：
- `task_id`
- 任务初始状态

---

## 4.2 任务启动阶段

**输入**：
- `task_id`

**后端处理**：
1. 状态更新为 `running`；
2. Orchestrator 读取任务配置，初始化上下文；
3. 生成首轮执行计划（Plan）。

**输出**：
- 运行实例 `run_id`
- 初始计划摘要

---

## 4.3 执行循环阶段（Act/Observe）

**后端处理**：
1. 根据计划决定调用哪个工具；
2. Tool Adapter 执行动作并返回观察结果；
3. 写入 `tool_call_log`（耗时、入参、结果、错误）；
4. 写入 `step_trace`（当前阶段与结论）。

**输出**：
- 每一步执行结果
- 证据（页面文本、截图链接、日志摘要）

---

## 4.4 异常与修正阶段（Reflect/Re-plan）

**触发条件**：
- 工具调用失败；
- 结果不满足任务目标；
- 连续失败达到阈值。

**后端处理**：
1. 异常分类（超时/权限/元素缺失/网络故障/策略错误）；
2. 执行重试策略（参数修正、延时重试、备用动作）；
3. 若仍失败，触发 Reflexion 生成新策略并重规划；
4. 超出上限则降级结束（给人工接管建议）。

**输出**：
- 修正后策略
- 重试记录
- 降级结论（如适用）

---

## 4.5 任务收敛与结果输出阶段（Deliver）

**后端处理**：
1. 聚合本次 run 的关键结论；
2. 计算核心指标（成功率、耗时、重试次数、工具成功率）；
3. 更新任务最终状态（`success` / `failed` / `degraded`）；
4. 生成结构化结果对象。

**输出**：
- 最终结论（TL;DR）
- 成功/失败状态
- 关键证据与建议动作
- 可回放 trace

---

## 5. 数据结构建议（最小集）

1. `task`
- `id`
- `task_type`
- `input_payload`
- `status`
- `created_at`

2. `run`
- `id`
- `task_id`
- `start_time`
- `end_time`
- `result_summary`

3. `step_trace`
- `run_id`
- `step_index`
- `phase`（plan/act/observe/reflect/replan/deliver）
- `decision`
- `observation`

4. `tool_call_log`
- `run_id`
- `tool_name`
- `input`
- `output`
- `success`
- `error_type`
- `duration_ms`

## 6. 接口清单建议（后端）

- `POST /api/tasks`：创建任务
- `GET /api/tasks`：任务列表
- `GET /api/tasks/{id}`：任务详情
- `POST /api/tasks/{id}/run`：启动任务
- `GET /api/runs/{run_id}`：运行摘要
- `GET /api/runs/{run_id}/trace`：轨迹详情
- `GET /api/runs/{run_id}/tool-calls`：工具调用日志
- `POST /api/runs/{run_id}/retry`：手动重试（可选）

---

## 7. 后端必须优先保证的能力

1. **幂等性**
- 重复请求不能创建重复运行实例。

2. **可观测性**
- 每次任务有唯一 Trace ID，日志可串联。

3. **错误可解释**
- 失败必须有明确错误类型和建议动作。

4. **可恢复性**
- 有重试与降级逻辑，避免一失败就终止。

5. **可评估性**
- 输出结构化数据，供任务二评估平台直接使用。

---

## 8. 与任务二评估平台对接点

后端需要额外保证输出以下评估字段：
- 任务输入与目标；
- 计划步骤数、重规划次数；
- 工具调用总数、成功率、平均耗时；
- 异常分类统计；
- 最终完成状态与结果质量摘要。

这些字段可直接支撑显式指标与过程评估。

---

## 9. 一句话总结

对 MyClaw 而言，后端不是“接口层”而是“控制中枢”：
- 负责任务编排、工具执行、异常修正、结果沉淀；
- 决定系统是否真正具备 Agent 的自治能力与可评估能力。
