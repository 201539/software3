# 软件工程项目助手 Agent 应用方案（迭代二）

## 1. 方案概述

本方案设计并实现一个面向软件工程团队协作场景的自主 Agent 应用，暂定名称为 **SE-Project Assistant**。该 Agent 目标是帮助团队完成从需求分解、任务规划、执行跟踪到风险预警的全流程协作支持。

相较于固定流程脚本，本应用强调 Agent 的“自治”能力：

- 能针对不同任务目标动态规划执行路径；
- 能调用外部工具与真实工程环境交互；
- 能在执行失败时进行异常处理与自我修正；
- 能可视化内部状态流转，便于解释与调试。

---

## 2. 应用定位与使用场景

### 2.1 面向对象

- 课程项目小组
- 中小型软件研发团队
- 需要快速建立项目管理智能化能力的开发者

### 2.2 典型使用场景

1. 输入高层需求后，自动拆解任务并生成里程碑计划；
2. 根据进度与阻塞信息，自动识别风险并提出调整建议；
3. 将任务同步到项目管理平台（如 GitLab Issue/看板）；
4. 每周自动生成项目周报（进度、风险、下一步计划）；
5. 对延期任务触发重规划并给出备选执行路径。

---

## 3. 功能需求（Agent 应用）

### 3.1 核心功能

1. **需求理解与任务分解**
   - 解析自然语言需求；
   - 识别功能点、依赖关系、优先级；
   - 输出可执行任务列表（支持负责人、预计工时、截止时间）。

2. **项目计划生成与动态调整**
   - 生成初始迭代计划与里程碑；
   - 依据任务状态变化进行重规划（例如：关键任务延期、成员请假）。

3. **风险识别与预警**
   - 自动检测风险信号（任务堆积、阻塞超过阈值、关键路径超时）；
   - 输出风险等级、影响范围与缓解策略。

4. **自动化协作输出**
   - 自动生成 Issue/任务卡片；
   - 自动生成每日报告/周报；
   - 提供可追溯的决策说明。

### 3.2 外部工具能力（至少 2 个）

本方案至少实现以下两个工具，可扩展第三个工具：

- **Tool A：GitLab 项目工具**
  - 功能：读取项目 Issue、创建/更新 Issue、读取 MR 状态；
  - 价值：将 Agent 规划结果落地到真实协作系统。

- **Tool B：日程与排期工具**
  - 功能：读取团队可用时间、安排里程碑时间点、检测冲突；
  - 价值：让计划具备可执行性，支持时间维度重规划。

- （可选）**Tool C：测试与质量状态工具**
  - 功能：读取 CI 状态、测试通过率、失败用例摘要；
  - 价值：让 Agent 能根据质量反馈调整优先级。

### 3.3 自我修正能力

- 识别工具调用失败（超时、参数错误、权限失败）；
- 执行重试与参数修复策略；
- 必要时切换降级方案（例如仅生成本地计划草案）；
- 记录异常事件并反馈给用户可解释的处理结果。

### 3.4 状态可视化能力

- 展示 Agent 执行轨迹：`目标 -> 计划 -> 行动 -> 观察 -> 反思 -> 重规划 -> 输出`；
- 展示工具调用日志与关键决策节点；
- 支持按任务查看执行历史与结果对比。

## 4. 推理框架设计

### 4.1 组合式推理框架（推荐）

为满足课程对 Agent 自治能力的要求，采用多框架协同：

1. **Plan-and-Execute（主框架）**
   - 先进行高层任务规划（需求分解、里程碑、风险点），再逐步执行；
   - 适合软件工程场景中的多阶段目标与动态调整。

2. **ReAct（执行框架）**
   - 在每个子任务执行阶段使用“思考-行动-观察”循环；
   - 结合外部工具（GitLab、日程、CI）进行实时决策与修正。

3. **Reflexion（反思框架，可选增强）**
   - 当出现连续失败、冲突信息或低质量输出时触发反思；
   - 基于失败原因更新策略并重规划，提升任务收敛能力。

### 4.2 框架路由策略

- 简单任务（如周报生成）：轻量 ReAct；
- 复杂任务（如迭代计划制定与落地）：Plan-and-Execute + ReAct；
- 连续异常任务（如多次工具失败或计划失效）：触发 Reflexion 重规划。

---

## 5. 技术架构设计

## 5.1 总体架构

采用“前端 + Agent 服务 + 工具层 + 数据层”的分层架构：

1. **前端层（Web）**
   - 功能：任务创建、执行监控、状态可视化、报告查看；
   - 技术建议：React/Vue + 组件库（如 Ant Design）。

2. **Agent 服务层（后端核心）**
   - 功能：意图解析、规划推理、行动决策、异常恢复；
   - 推理策略：可采用 ReAct + Plan-and-Execute（可加 Reflexion）。

3. **工具适配层（Tool Adapter）**
   - 功能：封装 GitLab、日程、CI 等外部系统调用；
   - 统一输入输出协议，便于新增工具。

4. **数据与日志层**
   - 功能：存储任务、执行日志、状态轨迹、错误记录；
   - 技术建议：MySQL/PostgreSQL + Redis（缓存/队列可选）。

## 5.2 关键流程

1. 用户输入目标（例如“为迭代二制定两周开发计划并同步到 GitLab”）；
2. Agent 先生成候选计划；
3. Agent 调用工具获取项目现状（Issue、成员负载、时间安排）；
4. Agent 比较计划与现实约束，执行修正；
5. Agent 执行落地动作（创建任务、设置截止日期）；
6. 输出结果并可视化全过程。

---

## 6. 非功能性需求

1. **安全性**
   - 严禁硬编码 API Key；
   - 使用环境变量或配置中心管理敏感信息；
   - 对关键操作增加权限控制与审计日志。

2. **可维护性**
   - 模块解耦（推理层、工具层、存储层）；
   - 提供统一错误码与日志规范。

3. **可扩展性**
   - 支持新增工具与新增任务模板；
   - 支持新增评估指标与策略。

4. **可观测性**
   - 提供结构化日志；
   - 提供任务级追踪 ID，便于复盘。

---

## 7. 迭代二交付范围建议（MVP）

### 7.1 必做交付

- Web 端可用的 Agent 应用；
- 至少 2 个可实际调用的外部工具（GitLab + 日程）；
- 自我修正机制（重试、降级、重规划）；
- 执行过程可视化页面；
- 演示视频可覆盖完整链路。

### 7.2 可选增强

- 引入多推理框架协同（如 ReAct + Reflexion）；
- 增加 CI 质量工具联动；
- 提供团队绩效与风险趋势图。

---

## 8. 与评估平台（任务二）的衔接

为便于后续评估平台接入，Agent 应用需要输出结构化评估数据：

- 输入任务与上下文；
- 计划步骤与修正次数；
- 工具调用序列、成功率、耗时；
- 最终结果与任务完成状态；
- 异常记录与恢复路径。

这些数据将直接支撑任务二中的显式指标与过程评估。

---

## 9. 风险与应对

1. **外部 API 不稳定**
   - 应对：重试机制 + 降级策略 + 缓存兜底。

2. **Agent 决策不稳定**
   - 应对：加入计划约束模板与反思环节，减少无效探索。

3. **演示链路过长导致不稳定**
   - 应对：准备标准演示数据与脚本，控制关键路径。

---

## 10. 总结

本方案以“软件工程项目助手”为主题，满足迭代二对 Agent 应用的核心要求：

- 具备自治规划与行动能力；
- 具备外部工具交互能力；
- 具备异常处理与自我修正能力；
- 具备可视化状态流转能力。

同时，采用前后端分离 Web 架构，便于与任务二评估平台联动，适合作为课程项目的主线实现方案。
