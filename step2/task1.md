# 迭代二任务一分析（Agent 应用）

## 1. 任务目标（你们要做什么）
搭建一个**面向特定领域**、具备“自主思考 + 自主行动”能力的 LLM Agent 应用。  
重点不是做一个固定流程脚本，而是做一个能够根据目标和环境动态决策的 Agent。

结合你们当前主题（软件工程项目助手），可以把目标落成：
- 帮助团队完成需求拆解、任务规划、进度跟踪、风险预警等软工协作任务；
- 能调用外部系统完成真实动作（不只是聊天回复）；
- 在失败时能自动恢复或降级。

---

## 2. 硬性需求拆解（逐条对应课程要求）

### 2.1 具备规划和思考能力（非预定义 workflow）
你们需要证明 Agent 会“先想后做”，而不是固定 if-else 流程。

建议实现：
- 一个显式的 Planning 阶段（生成计划步骤）；
- 执行中根据观察结果动态重规划（Plan -> Act -> Observe -> Replan）；
- 保留中间推理轨迹（至少结构化记录，不必暴露完整 CoT）。

可交付证据：
- 同一目标在不同输入约束下，生成不同执行路径；
- 出现阻塞后，Agent 自动调整步骤并继续执行。

### 2.2 至少 2 个主题相关外部工具
课程要求“与外部世界交互”，必须有工具调用，不可只做内存内 mock。

对你们主题的推荐工具：
1. **GitLab 工具**：读取/创建/更新 Issue、读取 MR/CI 状态；
2. **排期工具**：根据成员可用时间生成里程碑与截止日期。

可选第三工具：
- CI/测试状态工具（质量反馈驱动优先级调整）。

验收关键点：
- 有统一 Tool 接口；
- 每次调用有入参、出参、耗时、状态日志；
- 工具失败有捕获和回退。

### 2.3 自我修正能力
系统异常不能直接崩溃，需要“发现问题 -> 修复尝试 -> 失败再降级”。

建议最小实现：
- 重试策略（超时/临时失败）；
- 参数修复（例如字段缺失时自动补全默认值）；
- 降级策略（外部系统不可用时先产出本地计划草案）；
- 错误可解释反馈（告诉用户发生什么、系统做了什么补救）。

### 2.4 内部状态流转可视化
课程明确要求可视化（CLI/Web 均可）。

建议页面或面板展示：
- 状态机轨迹：`Goal -> Plan -> Action -> Observation -> Reflection -> Output`；
- 每步工具调用记录与结果；
- 异常和重试过程。

---

## 3. 推荐技术实现路线（MVP）

### 3.1 架构分层
- 前端（Web）：任务输入、执行过程可视化、结果展示；
- 后端 Agent 服务：规划器 + 执行器 + 反思/重规划模块；
- Tool Adapter 层：封装 GitLab、排期工具；
- 存储层：任务记录、调用日志、状态轨迹。

### 3.2 推理范式建议（可拿加分）
课程提到可加分：多推理框架组合。  
建议采用：
- 主流程：ReAct（边思考边行动）；
- 长任务：Plan-and-Execute（先全局计划，再逐步执行）；
- 出错后：Reflexion（基于失败反思再尝试）。

## 4. 迭代二任务一的交付清单（建议按此准备）
1. 可运行 Agent 应用（最好 Web 形态，便于演示）；
2. 至少 2 个真实可调用工具；
3. 自我修正机制（重试/参数修复/降级至少其二）；
4. 状态可视化界面（或 CLI 过程面板）；
5. 演示视频（5-10 分钟）覆盖完整链路；
6. 文档说明架构设计、核心流程和异常处理方案。

---

## 5. 常见风险与规避
- **误区 1：做成固定流程机器人** -> 必须体现动态重规划；
- **误区 2：工具只是 mock** -> 至少两个真实外部系统交互；
- **误区 3：只展示结果不展示过程** -> 增加状态流与调用日志；
- **误区 4：把 API Key 写进代码** -> 使用环境变量/配置注入。
