# 迭代一评估报告

## 1. 评估指标选取原因

本次评估围绕外卖 Agent 任务轨迹，选择了 4 类互补指标，覆盖"动作正确性—规划合理性—任务完成度"的完整链路：

1. **工具调用准确度（tool_call_accuracy）**
   - 评估 Agent 工具调用是否与期望步骤一致，强调**顺序一致**与**参数精确匹配**。
   - 适用于定位"工具选错/参数错/顺序错"的关键问题。
   - 参考 Ragas `ToolCallAccuracy` 指标复现，采用相同的序列对齐 + 参数逐键比较方式。

2. **工具调用 F1（tool_call_f1）**
   - 评估工具调用集合层面的覆盖程度（不强制顺序）。
   - 用于衡量"该调用的是否都调用到了、乱调用是否多"。
   - 参考 Ragas `ToolCallF1` 指标复现，将工具调用转为可哈希集合后计算 Precision/Recall/F1。

3. **规划合理性（plan_quality）**
   - 从轨迹结构层面评估规划问题，检测**无效工具调用、死循环、逻辑断层、缺失工具、冗余步骤**。
   - Ragas 无直接对应指标，此为基于外卖 Agent 场景设计的自定义指标。

4. **任务完成度（task_completion，LLM 判别）**
   - 直接评估最终回复是否满足用户诉求与期望结果。
   - 通过 LLM 输出 `completed` + `missing`（缺失要素）实现可解释性判断。
   - 参考 Ragas `AgentGoalAccuracy` 的 LLM-as-Judge 思路，采用二值判定。

上述指标互补：
- `tool_call_accuracy` 极严格（顺序+参数），适合"硬正确性"评估；
- `tool_call_f1` 偏召回视角，不惩罚顺序；
- `plan_quality` 评估轨迹结构质量；
- `task_completion` 面向最终用户体验。

## 2. 指标实现过程（核心逻辑）

### 2.1 工具调用准确度

- 若预测/参考均为空，记 1.0；任一为空则 0.0。
- 序列对齐（默认 strict_order=True）：工具调用序列完全一致才给 `aligned=1`。
- 非严格模式下，先按工具名+参数排序再配对（与 Ragas `sorted_key_for_tool_call` 一致）。
- 对齐后逐步比较参数：只检查参考参数中出现的 key，值转为字符串后比较（与 Ragas `exact_match_args` 中 `str()` 转换一致）。
- 评分公式：

  $$score = avg\_arg\_score \times coverage \times aligned$$

### 2.2 工具调用 F1

- 将每步工具调用（工具名 + 参数）转为可哈希元组，构建集合。
- 计算 TP/FP/FN，输出 F1。
- 不强制顺序，体现"覆盖率"。

### 2.3 规划合理性

基于外卖 Agent 场景的工具依赖链，设计了 5 项检测：

| 检测项 | 说明 | 每项扣分 |
|--------|------|---------|
| 无效工具 | 调用了系统中不存在的工具（如 `get_menu`） | -0.15 |
| 死循环 | 相邻步骤调用同一工具（错误重试模式） | -0.10 |
| 逻辑断层 | 跳过前置依赖（如未 `search_products` 就直接 `place_order`） | -0.10 |
| 缺失工具 | 期望工具未出现在实际调用中 | -0.10 |
| 冗余步骤 | 实际步骤数多于期望步骤数 | -0.05 |

工具依赖链定义：
```
search_restaurants → search_products → place_order → pay_order
```

### 2.4 任务完成度（LLM 判别 + 缺失要素）

- 使用 LLM（glm-4-flash）对 `user_query + ground_truth + final_answer` 进行判别。
- 要求 LLM 输出 JSON：
  ```json
  {"completed": 0或1, "reason": "...", "missing": ["要素1", "要素2"]}
  ```
- 规则增强：**只要 `missing` 非空，强制 `completed=0`**，避免"缺失但仍完成"的情况。
- 若 LLM 调用失败，回退到规则版关键词匹配。

## 3. 评估结果分析

从 `step1/reports/eval_report.md` 汇总结果（27 个样本）：

| 指标 | 均值 |
| --- | --- |
| tool_call_accuracy | 0.4537 |
| tool_call_f1 | 0.6987 |
| task_completion | 0.9259 |
| plan_quality | 0.9093 |

### 3.1 主要观察

1. **工具调用准确度偏低（0.4537）**
   - 说明严格顺序+参数对齐时，错误较多。
   - 常见原因：工具顺序不一致、参数值错误、遗漏步骤（如 9 个样本缺少 `pay_order`）。
   - 该指标与 Ragas 实现一致：序列不完全匹配时 `aligned=0`，整体得分归零。

2. **工具调用 F1 较高（0.6987）**
   - 表明大部分核心调用"被覆盖"，但顺序和参数仍有问题。
   - 说明 Agent "知道该用什么工具"，但执行细节不足。

3. **规划合理性（0.9093）具有区分度**
   - 完美样本得到 1.0，有问题的样本被有效扣分：
     - 样本 e5f6a7b8c9d01234：3 次错误-重试死循环，得分 **0.55**
     - 样本 d4e5f6a7b8c90123：调用不存在的 `submit_payment`，得分 **0.70**
     - 样本 c3d4e5f6a7b89012：调用不存在的 `get_menu`，得分 **0.80**
     - 样本 d377268ae116c741：`place_order` 在 `search_products` 之前（逻辑断层），得分 **0.85**
   - 检测到的问题类型：无效工具调用 2 例、死循环 4 例、逻辑断层 1 例。

4. **任务完成度（0.9259）**
   - 25 个样本完成度为 1.0，2 个样本为 0.0（缺失配送状态/支付方式等关键要素）。
   - 采用与 Ragas `AgentGoalAccuracy` 一致的二值判定方式。

---

## 4. 结论

目前指标实现已覆盖工具调用正确性、规划合理性和任务完成度三个层面。整体表现显示：
- Agent 工具调用覆盖率较高（F1=0.70），但严格准确度不足（Accuracy=0.45），主要因为遗漏支付步骤和参数值偏差；
- 规划结构总体合理（0.91），但存在个别样本的死循环和逻辑断层问题；
- 任务完成度较高（0.93），LLM 判别配合 `missing` 强制规则能有效识别关键要素缺失。
