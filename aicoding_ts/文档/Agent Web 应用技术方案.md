# AI Coding Agent Web 应用技术方案（Agent First 版）

## 1. 项目目标

本项目是一款面向全栈工程师的轻量级 AI Coding Agent Web 应用。

它的核心不是做成传统意义上的前后端分离 Web 系统，而是做成一个以 Agent 为中心的代码工作台：

- 用户通过 Web 进行对话和查看结果
- Agent 负责理解需求、规划任务、调用工具、修改工作区
- 工作区保存真实代码和项目状态
- 命令行和工具执行是核心能力之一
- Web 只是 Agent 的可视化入口和操作入口

### 产品形态

- Chat
- 代码编辑器
- 工作区文件列表
- Agent 执行状态与日志
- 预览与结果回显

### 核心目标

- 用户输入需求后，系统能生成项目骨架
- 用户可以继续对话让 Agent 修改项目
- 用户能看到工作区文件结构和代码内容
- Agent 能通过受控工具完成读写文件、运行命令、检查结果
- 产品保持轻量、中文友好、适合全栈工程师使用

---

## 2. 设计原则

### 2.1 Agent First

系统的主导逻辑不是页面路由或 API 组织，而是任务流和 Agent 执行流。

### 2.2 工具驱动

Agent 不能直接“猜代码”，必须通过工具读取上下文、修改文件、运行检查、查看结果。

### 2.3 工作区优先

真实代码始终保存在工作区中，所有生成、修改、预览、回滚都围绕工作区展开。

### 2.4 最小通信约束

系统内部不强制采用传统前后端 API 作为中心架构。

- Web 和系统之间可以有必要的交互通信
- 内部优先使用任务、事件、流式回显、工具调用、CLI 执行结果回传
- API 可以存在，但不是核心架构中心

### 2.5 先可用，再扩展

MVP 阶段优先保证主流程闭环：

1. 输入需求
2. 生成项目骨架
3. 展示文件树和编辑器
4. 继续对话修改
5. 运行检查与回显结果

---

## 3. 用户场景

### 3.1 创建项目

用户输入一句自然语言需求，Agent 生成一个可继续开发的项目骨架。

### 3.2 持续修改

用户在对话框里继续提出需求，Agent 根据当前工作区内容继续修改。

### 3.3 查看与编辑

用户可以浏览工作区文件列表，打开文件，查看代码，必要时手动编辑。

### 3.4 执行与验证

Agent 可以在受控环境中执行命令，完成安装、检查、构建、测试、预览等动作。

---

## 4. 总体架构

建议采用以下五层架构：

1. **Web 交互层**
2. **Agent 编排层**
3. **工具执行层**
4. **工作区与运行环境层**
5. **状态与记录层**

### 4.1 Web 交互层

Web 只负责交互和展示，不负责核心决策。

主要包含：
- Chat 面板
- 文件树面板
- 代码编辑器
- 执行状态面板
- diff 预览
- 预览结果区域

### 4.2 Agent 编排层

这是系统的大脑，负责：
- 理解用户需求
- 生成任务计划
- 选择上下文
- 决定调用哪些工具
- 推进任务状态
- 汇总修改结果

### 4.3 工具执行层

负责受控地执行真实动作：
- 读文件
- 列目录
- 写文件
- 搜索文本
- 运行命令
- 创建快照
- 回滚版本

### 4.4 工作区与运行环境层

负责实际承载项目代码和执行结果：
- 独立工作区目录
- 受控 shell / CLI
- 构建运行环境
- 预览服务
- 日志输出

### 4.5 状态与记录层

负责保存系统运行过程中的关键状态：
- 项目状态
- 会话状态
- Agent 任务状态
- 文件索引
- diff 历史
- 版本快照
- 工具调用日志

## 5. 核心模块设计

## 5.1 Web UI 模块

Web UI 需要满足：
- 低认知负担
- 快速查看工作区
- 清晰展示 Agent 正在做什么
- 让用户能及时介入、确认或修改

### 建议布局

- 左侧：工作区文件树
- 中间：代码编辑器
- 右侧或底部：Chat + Agent 状态 + 日志

## 5.2 Agent Orchestrator

Agent Orchestrator 是任务调度中心，也是整个 Agent 编排层的唯一入口。

它负责把一次用户请求拆成多个可执行阶段，并决定每个阶段由哪个角色处理、何时调用工具、何时停止、何时返回给用户确认。

它的职责：
- 接收用户输入
- 判断当前任务阶段
- 生成计划
- 调用工具
- 记录执行轨迹
- 判断是否继续、重试或结束
- 控制 Planner、Executor、Reviewer、Summarizer 的协同顺序

### 5.2.1 编排方式

建议采用“规则 + LLM”混合编排，而不是纯规则或纯模型。

- **规则负责**：状态流转、工具权限、失败重试、终止条件、危险操作拦截
- **LLM 负责**：任务理解、任务拆解、代码修改策略、结果解释、总结表达

也就是说，Orchestrator 用规则控制流程边界，用 LLM 完成需要语义理解和生成的部分。

### 5.2.2 四个角色的职责

#### Planner
Planner 负责把用户需求转换成结构化任务计划。

主要产出：
- 目标
- 任务步骤
- 需要读取的文件
- 需要修改的文件
- 风险点
- 是否需要用户确认

Planner 一般由 LLM 驱动，但必须输出结构化结果，不能只输出自由聊天文本。

#### Executor
Executor 负责根据 Planner 的计划执行具体改动。

主要行为：
- 读取上下文
- 调用工具
- 修改文件
- 运行命令
- 保存版本快照

Executor 既依赖 LLM 做代码生成和补丁生成，也依赖工具层完成真实文件操作。

#### Reviewer
Reviewer 负责检查 Executor 的结果是否可接受。

主要检查内容：
- 代码是否符合目标
- 是否引入明显错误
- 是否需要补充文件
- 是否需要再次修复
- 是否可以进入总结阶段

Reviewer 可以是 LLM Review，也可以叠加规则检查，例如 lint、test、build 结果。

#### Summarizer
Summarizer 负责把本轮任务结果转换成用户可读的反馈。

主要产出：
- 做了什么
- 改了哪些文件
- 验证结果如何
- 是否成功
- 下一步建议

Summarizer 通常由 LLM 生成自然语言总结，但必须基于真实工具结果和 diff，不允许凭空编造。

### 5.2.3 协同流程

推荐的协同顺序如下：

1. **Planner 规划**
   - 输入：用户需求 + 当前上下文
   - 输出：结构化任务计划

2. **Orchestrator 校验计划**
   - 检查计划是否越权
   - 检查计划是否超出本轮范围
   - 检查是否需要用户确认

3. **Executor 执行**
   - 按计划读取上下文
   - 调用工具修改文件
   - 必要时运行命令

4. **Reviewer 检查**
   - 读取 diff、日志、构建结果
   - 判断是否通过
   - 若未通过，返回修复建议或触发重试

5. **Summarizer 汇总**
   - 生成用户可读摘要
   - 输出变更文件、结果和下一步

### 5.2.4 谁来调度

调度者不是某个独立的 LLM 角色，而是 **Agent Orchestrator 本身**。

Orchestrator 是确定性的控制层，负责：
- 调度 Planner
- 调度 Executor
- 调度 Reviewer
- 调度 Summarizer
- 管理任务状态机
- 决定是否进入下一阶段

LLM 只是被 Orchestrator 调用来完成某些子任务，不是系统的最高控制者。

### 5.2.5 是否基于 LLM

四个角色并不是都必须是独立模型实例，但它们的核心智能部分通常都由 LLM 提供。

推荐实现方式：
- Planner：LLM + 结构化输出
- Executor：LLM 生成修改建议 + 工具层执行
- Reviewer：LLM 评审 + 规则校验 + 自动测试结果
- Summarizer：LLM 总结 + 任务日志约束

### 5.2.6 规则和模型的边界

#### 规则层负责
- 状态机流转
- 工具权限控制
- 命令白名单
- 超时与重试
- 版本快照
- 危险操作确认

#### 模型层负责
- 理解自然语言需求
- 生成计划
- 推断上下文相关性
- 生成代码修改方案
- 总结结果

### 5.2.7 失败与重试策略

如果 Reviewer 发现问题，Orchestrator 可以：
- 直接触发 Executor 修复
- 让 Planner 重新规划
- 请求用户确认
- 终止本轮并回滚版本

重试不能无限进行，建议设置最大重试次数，避免 Agent 在错误循环里消耗资源。

这些角色可以是逻辑上的，不一定是多个独立服务。

## 5.3 Context Builder

负责把“当前项目里最相关的信息”组织给模型，并且显式控制 token 预算。

Context Builder 不是简单地“把所有信息拼起来”，而是一个 **检索 + 启发式筛选 + 摘要压缩 + 预算分配** 的组合系统。

### 5.3.1 输入来源

- 用户当前请求
- 文件树
- 当前打开文件
- 最近变更 diff
- 最近修改过的文件
- 错误日志
- 项目摘要
- 文件统计信息（大小、修改时间、引用次数、是否高频打开）
- 向量检索结果（可选）

### 5.3.2 上下文选择策略

Context Builder 推荐采用 **“启发式优先，检索增强兜底”** 的策略，而不是完全依赖单一 RAG。

#### 第一层：强相关启发式
优先选入以下信息：
- 当前打开文件
- 最近修改的文件
- 当前任务直接命中的文件
- 报错栈中提到的文件
- 路由、组件、服务、配置等关键入口文件

这层的目标是保证 Agent 在大多数常见场景下，都能快速拿到最有用的信息。

#### 第二层：结构化扩展
如果 token 预算允许，再扩展到：
- 同目录相关文件
- 被当前文件 import / 引用的文件
- 同一功能模块下的其他文件
- 与当前任务关键词高匹配的文件

#### 第三层：RAG / 向量检索兜底
当项目较大、文件众多、仅靠启发式无法准确定位时，再使用 RAG 或向量检索。

适用场景：
- 用户描述较抽象
- 目标文件不明显
- 项目结构复杂
- 跨模块修改

推荐原则：
- **小项目：启发式优先，RAG 可选**
- **大项目：启发式 + RAG 混合**
- **不要一上来全量 RAG**，否则成本高、噪音大、调试困难

### 5.3.3 文件相关性评分

可以给每个文件打一个相关性分，按分数决定是否进入上下文。

建议评分维度：
- 最近是否打开
- 最近是否修改
- 是否被当前文件引用
- 路径是否与任务关键词匹配
- 文件类型是否关键（如路由、组件、配置、入口、服务）
- 错误日志是否命中
- 向量相似度是否高（如果启用 RAG）

示例：
- 当前打开文件：最高优先级
- 最近修改文件：高优先级
- 报错命中文件：高优先级
- 同模块文件：中优先级
- 低相关工具文件：低优先级

### 5.3.4 Token 预算控制

Context Builder 必须有明确的 token 预算，而不是无限制拼接。

建议把 token 预算分成几个槽位：
- **任务摘要预算**：放用户需求、目标、约束
- **项目摘要预算**：放项目结构和技术栈
- **关键文件预算**：放最相关文件内容
- **最近变更预算**：放 diff / patch
- **错误日志预算**：放必要报错信息
- **回复预算**：留给模型输出本轮结果

### 5.3.5 长文件处理策略

当文件很大时，不能直接整篇塞进上下文，建议采用分层摘要。

#### 方案一：按段落/函数切片
- 先按函数、类、组件、模块切片
- 只取与当前任务相关的切片
- 无关切片不进入上下文

#### 方案二：先摘要，再补原文
- 先对长文件生成摘要
- 再根据需要补充关键片段
- 适合配置文件、服务文件、页面文件

#### 方案三：分层摘要
- **一级摘要**：文件整体在做什么
- **二级摘要**：关键函数/组件/接口做什么
- **三级原文**：只保留真正会被修改的片段

#### 方案四：按变更点定位
如果 Agent 只需要改某个局部区域，可以只提取：
- 与目标相关的函数
- 报错行附近代码
- import 区域
- 当前 diff 附近上下文

### 5.3.6 何时使用 RAG

RAG 不是默认全局开启，而是按需启用。

适合使用 RAG 的情况：
- 项目很大
- 文件很多
- 任务描述比较模糊
- 仅靠最近修改和当前文件无法定位相关代码
- 需要跨模块查找历史实现

不适合过度依赖 RAG 的情况：
- 项目很小
- 目标文件很明确
- 本轮只做局部修改
- 上下文窗口本来就够用

### 5.3.7 推荐的上下文构建流程

1. 读取用户请求
2. 生成任务关键词和意图标签
3. 从文件树和历史状态中筛选高优先级文件
4. 对长文件做摘要或切片
5. 如有必要，调用 RAG 检索补充候选文件
6. 根据 token 预算裁剪上下文
7. 输出最终上下文包给 Planner / Executor

### 5.3.8 输出内容

Context Builder 最终输出的不是“越多越好”的文本，而是一个可控的上下文包，通常包含：
- 当前任务摘要
- 相关文件列表
- 相关文件片段或摘要
- 最近变更摘要
- 报错摘要
- 预算使用情况

### 5.3.9 设计目标

Context Builder 的目标是：
- 让模型拿到“刚刚好”的信息
- 避免上下文过载
- 避免检索噪音
- 避免因为信息不足导致胡猜
- 让大项目和长文件也能稳定工作

## 5.4 Tool Gateway

Tool Gateway 是所有执行动作的统一出口。

它负责：
- 参数校验
- 路径校验
- 命令执行安全控制
- 权限限制
- 超时控制
- 日志审计
- 错误处理

### 5.4.1 为什么不能只靠命令白名单

对于 coding agent 来说，命令并不是固定几条，而是会随着任务动态变化，例如：
- `npm install express`
- `python test.py`
- `pnpm add zod`
- `pytest -q`
- `npm run build`

如果只做静态白名单，会过于死板，无法覆盖真实开发场景。
如果完全放开，又会有命令注入、危险操作和资源滥用风险。

因此需要的是 **“受控的动态命令执行”**，而不是简单的允许 / 禁止二选一。

### 5.4.2 命令执行安全模型

建议采用“**动态生成 + 结构化校验 + 风险分级 + 沙箱执行 + 结果审计**”的组合方案。

#### 第一层：上下文驱动的命令生成

命令由 Agent 根据当前任务动态生成，但必须来自任务上下文，而不是用户原始文本直接拼接到 shell。

例如：
- 用户说“安装 express”
- Agent 生成 `npm install express`
- 但不是把用户原话直接当 shell 执行

这一步可以减少明显的注入风险。

#### 第二层：命令结构化解析

命令在执行前先被解析为结构化结构，而不是纯字符串直接执行。

建议解析字段包括：
- 基础命令：`npm` / `python` / `pnpm` / `pytest`
- 子命令：`install` / `run` / `test`
- 参数列表
- 工作目录
- 是否需要联网
- 是否会写文件
- 是否可能删除文件
- 是否可能覆盖安装

解析后先判断“命令意图”，再决定是否执行。

#### 第三层：风险分级

不是所有命令都一样危险，建议按风险分级处理。

##### 低风险
- `npm install express`
- `python test.py`
- `pytest -q`
- `npm run build`
- `pnpm add zod`

这类命令通常属于正常开发操作，可以在受控环境中直接执行。

##### 中风险
- 会联网拉包
- 会修改大量文件
- 会生成新项目骨架
- 会触发长时间构建

这类命令可以执行，但要有更严格的超时、日志和资源限制。

##### 高风险
- 删除目录
- 格式化整仓库
- 覆盖写大量文件
- 修改系统级配置
- 访问工作区外部路径
- 可疑 shell 拼接命令

这类命令需要用户确认，或者必须由更严格的规则拒绝。

#### 第四层：参数安全校验

在执行前必须对参数做安全处理：
- 禁止直接拼接未转义的用户输入
- 禁止将包含 `;`、`&&`、`|`、反引号、命令替换等可疑片段的字符串直接透传到 shell
- 对路径参数做归一化和越界检查
- 对工作目录做强制限制，不能跳出项目工作区
- 对环境变量做白名单过滤

#### 第五层：沙箱执行

命令必须在受控执行环境中运行，例如：
- Docker 容器
- 隔离工作目录
- 受限权限账户
- 网络受控或可配置网络
- CPU / 内存 / 时间限制

这样即使命令本身有问题，也不会影响宿主环境。

#### 第六层：结果审计

每次命令执行后都要记录：
- 命令内容
- 触发来源
- 工作目录
- 开始和结束时间
- exit code
- stdout / stderr 摘要
- 影响文件
- 是否需要重试

### 5.4.3 推荐的安全策略

建议采用以下组合，而不是单一白名单：

1. **允许动态命令，但限制执行上下文**
   - 只允许在工作区内执行
   - 只允许由 Agent 通过工具调用触发

2. **保留命令模板级白名单**
   - 例如 `npm install <pkg>`、`python <script>`、`npm run <script>`、`pytest <args>`
   - 重点不是固定命令字符串，而是允许哪些命令模板和参数模式

3. **引入风险评分**
   - 低风险自动执行
   - 中风险加强限制
   - 高风险需要确认或拒绝

4. **引入动态策略判断**
   - 根据当前任务、历史行为、项目状态、命令参数判断是否放行
   - 不是只看静态字符串是否在名单中

5. **默认最小权限**
   - 不给 shell 超出任务所需的权限
   - 不允许访问工作区外路径
   - 不允许任意系统命令

### 5.4.4 可以支持的命令类型

在 MVP 阶段，建议支持以下几类常见开发命令：
- 包管理：`npm install <pkg>`、`pnpm add <pkg>`、`pip install <pkg>`
- 运行脚本：`npm run <script>`、`python <file>`、`pytest <args>`
- 构建检查：`npm run build`、`npm run lint`
- 生成项目：模板脚手架命令

### 5.4.5 何时必须阻止或确认

以下情况应阻止或要求用户确认：
- 命令包含明显注入风险
- 命令试图访问工作区外目录
- 命令会删除大量文件
- 命令会覆盖用户未确认的修改
- 命令长时间运行且影响资源
- 命令试图做系统级改动

### 主要工具

- `list_files`
- `read_file`
- `write_file`
- `patch_file`
- `search_in_workspace`
- `run_command`
- `create_snapshot`
- `restore_snapshot`
- `read_logs`

## 5.5 Workspace Manager

负责管理每个项目的真实工作区。

建议采用：
- 每个项目独立目录
- 每个项目独立快照目录
- 每个项目独立日志目录

它要支持：
- 创建项目工作区
- 初始化模板
- 记录文件变更
- 创建版本快照
- 回滚到某个状态

## 5.6 Preview Runner

负责启动、维护和暴露预览结果。

它要支持：
- 构建项目
- 运行本地预览
- 输出错误日志
- 提供预览 URL

## 6. Agent 工作流程

### 6.1 标准任务流

1. **接收需求**
   - 用户输入自然语言需求

2. **生成计划**
   - Agent 拆解任务
   - 确定本轮目标
   - 确定要读哪些文件
   - 确定要改哪些文件

3. **检索上下文**
   - 读取文件树
   - 读取相关文件
   - 读取最近 diff
   - 读取错误日志

4. **执行修改**
   - 通过工具写文件或 patch 文件
   - 记录变更

5. **运行验证**
   - 执行 lint / test / build / 启动预览
   - 收集输出

6. **总结回显**
   - 输出本轮改了什么
   - 哪些文件受影响
   - 是否成功
   - 下一步建议是什么

### 6.2 失败处理

如果验证失败，Agent 需要：
- 读取错误信息
- 分析失败原因
- 尝试局部修复
- 必要时请求用户确认

---

## 7. Agent 的基本能力保证方式

### 7.1 规划能力

通过“先计划后执行”的强制流程保证：
- 所有复杂任务先拆步骤
- 每轮只处理一个明确目标
- 计划输出结构化

### 7.2 工具使用能力

通过 Tool Gateway 保证：
- 工具能力显式注册
- 参数强校验
- 权限可控
- 所有调用留痕

### 7.3 上下文管理能力

通过 Context Builder 保证：
- 只提供相关内容
- 对长内容摘要压缩
- 最近变更优先
- 当前任务优先

### 7.4 结果汇总能力

通过 Summary Generator 保证：
- 每轮执行后都输出结构化总结
- 总结基于真实 diff 和工具结果
- 成功和失败都要可读、可追踪

---

## 8. 通信与交互方式

### 8.1 不以 API 为核心

系统内部不需要把所有能力都设计成传统 REST API。

更适合的方式包括：
- 任务消息
- 事件流
- SSE / WebSocket 流式输出
- CLI 命令执行结果回传
- 内部模块函数调用

### 8.2 仍然可能存在的必要通信

Web 层仍需要和系统进行必要的交互，例如：
- 发起任务
- 获取状态
- 拉取文件内容
- 保存编辑
- 获取执行日志

但这不意味着系统要回到传统前后端中心化 API 架构。

### 8.3 MVP 推荐

MVP 阶段可以采用：
- WebSocket 或 SSE 做流式输出
- 任务消息驱动 Agent 状态变化
- 工具调用通过内部网关完成

---

## 9. 数据与状态设计

### 9.1 需要保存的核心状态

- 用户
- 项目
- 会话
- 消息
- 文件索引
- 任务状态
- 工具调用记录
- 版本快照
- 构建日志

### 9.2 数据存储原则

- MVP 阶段先保证可用
- 数据可以先轻量化存储
- 只在需要时引入更完整的数据库和对象存储
- 不把数据库当成架构中心，而是状态持久化手段

---

## 10. 工作区与版本策略

### 10.1 工作区策略

- 每个项目一个独立工作区
- Agent 只在当前项目工作区内操作
- 不能跨项目越权访问

### 10.2 版本策略

- 每轮重要修改后生成快照
- 支持 diff 查看
- 支持回滚
- 支持恢复到某一轮任务前状态

---

## 11. 安全策略

这是 Agent Web 应用必须重视的部分。

### 11.1 主要风险

- prompt 注入
- 文件越权访问
- 命令注入
- 恶意代码执行
- 资源耗尽
- 敏感信息泄露

### 11.2 防护手段

- 工作区隔离
- 命令白名单或命令级限权
- 路径限制
- 超时限制
- 资源限制
- 高风险动作二次确认
- 所有工具调用留痕

---

## 12. 可观测性设计

系统必须能回答三个问题：

1. Agent 做了什么？
2. 为什么这么做？
3. 做完之后结果如何？

### 12.1 必须记录

- 用户请求
- 任务计划
- 工具调用
- 文件 diff
- 命令输出
- 错误日志
- 最终总结

### 12.2 关键指标

- 首次生成成功率
- 修改成功率
- 构建成功率
- 平均任务耗时
- 重试率
- 回滚率

---

## 13. MVP 技术落地建议

### 13.1 最小可行架构

MVP 建议做成一个轻量的单体系统，但内部按 Agent 架构拆分，而不是按传统前后端拆分。

建议包含：
- Web UI
- Agent Orchestrator
- Tool Gateway
- Context Builder
- Workspace Manager
- Preview Runner
- State Store

### 13.2 开发顺序

1. 搭工作区和文件操作能力
2. 做 Agent 任务流
3. 做上下文构建和工具调用
4. 做 Chat 和文件树 UI
5. 做代码编辑器和保存
6. 做版本快照与回滚
7. 做预览和验证

---

## 14. 为什么这个方案适合你的产品

这个方案的优势是：

- 更符合 Agent 软件本质
- 不被传统前后端架构限制
- 更适合命令行式执行任务
- 更适合“生成骨架 + 持续修改”的主流程
- 更容易先做出 MVP
- 更方便后续再拆分和扩展

---

## 15. 最终总结

这是一个以 Agent 为中心的轻量级 AI Coding Web 应用技术方案。系统不再以传统前后端分层和 API 驱动为核心，而是以任务流、工具调用、上下文管理、工作区操作和版本回滚为主线，Web 只是交互入口，Agent 才是整个系统真正的执行中枢。
