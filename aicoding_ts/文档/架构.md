# AI Coding Agent Web 软件架构说明

## 1. 架构目标

本项目采用 **Agent-first** 的架构思路，而不是传统前后端强分层架构。

核心目标是：

- 让 Agent 能直接理解任务、规划步骤、调用工具、修改工作区、验证结果并汇总输出
- 让 Web 只是一个轻量交互入口，而不是系统的主逻辑中心
- 让架构围绕“任务执行闭环”来设计，而不是围绕“接口 CRUD”来设计
- 保持轻量、灵活、适合中文用户，并适配从 0 开始的 MVP

---

## 2. 架构原则

### 2.1 Agent 优先

系统的主逻辑由 Agent 驱动，核心流程是：

1. 理解用户需求
2. 规划任务
3. 构建上下文
4. 调用工具
5. 修改工作区
6. 执行验证
7. 汇总结果

### 2.2 Web 只是交互壳

Web 层只负责：

- 输入需求
- 展示聊天过程
- 展示代码编辑器
- 展示文件列表
- 展示任务状态、diff 和日志

Web 不负责核心决策，也不直接操作工作区。

### 2.3 工具受控执行

所有文件修改、命令执行、目录读取、日志读取，都必须通过受控工具层完成，不能让模型直接接触底层资源。

### 2.4 按需引入存储

数据库、缓存、任务队列、版本快照等能力都可以按需加入，不要求一开始就全量搭建。

### 2.5 不强制传统前后端分层

系统内部可以采用：

- 函数调用
- 模块协作
- 事件流
- WebSocket / SSE
- CLI 执行回传
- 本地任务调度

API 可以存在，但不作为唯一主导方式。

---

## 3. 总体架构

可以把系统理解为五层：

1. **交互层**：Chat、编辑器、文件树、日志、任务状态
2. **Agent 层**：规划、上下文管理、工具决策、总结
3. **工具层**：文件操作、命令执行、日志读取、回滚
4. **执行环境层**：工作区、终端、沙箱、预览环境
5. **状态层**：任务状态、对话记录、diff、版本、索引

### 3.1 架构示意

```text
用户
  ↓
Web UI（Chat / 编辑器 / 文件树 / 日志）
  ↓
Agent Orchestrator（任务编排器）
  ↓
Context Builder（上下文构建）
  ↓
Model Router / LLM 调用
  ↓
Tool Gateway（受控工具网关）
  ↓
Workspace / Sandbox / Shell / File System
  ↓
结果、diff、日志、状态回传
  ↓
Web UI 展示
```

## 4. 核心模块设计

### 4.1 Web UI 层

#### 职责

- 接收用户输入
- 展示 AI 对话过程
- 展示代码编辑器
- 展示工作区文件列表
- 展示任务执行状态
- 展示 diff、错误和日志

#### 特点

- 轻量
- 简洁
- 中文友好
- 不承载复杂业务逻辑

---

### 4.2 Agent Orchestrator

这是整个系统的大脑。

#### 职责

- 接收用户任务
- 拆解任务
- 决定是否需要读文件、改文件、跑命令
- 调度工具
- 维护任务状态
- 判断是否继续执行、重试或结束
- 输出总结结果

#### 推荐拆分

- **Planner**：生成执行计划
- **Executor**：执行文件修改与命令调用
- **Reviewer**：检查输出是否正确
- **Summarizer**：生成对用户可读的结果总结

---

### 4.3 Context Builder

#### 职责

- 读取目录树
- 选择与任务最相关的文件
- 摘要长文件
- 整理最近变更
- 组装给模型的上下文包

#### 目标

避免把整个项目强塞进上下文，而是让模型只看到“当前任务真正需要的内容”。

---

### 4.4 Model Router

#### 职责

- 统一管理模型调用
- 选择不同模型处理不同任务
- 支持规划、代码修改、总结、检查等不同场景

#### 设计建议

可以根据任务类型路由：

- 规划任务 → 更擅长推理的模型
- 代码编辑 → 更擅长编码的模型
- 摘要总结 → 便宜且稳定的模型

---

### 4.5 Tool Gateway

这是保证 Agent 可控的关键层。

#### 职责

- 暴露工具能力给 Agent
- 校验参数
- 限制路径和命令
- 记录审计日志
- 处理超时、失败和重试

#### 常见工具

- `read_file`
- `list_files`
- `write_file`
- `create_file`
- `delete_file`
- `search_text`
- `run_command`
- `read_logs`
- `rollback_file`
- `snapshot_workspace`

#### 安全原则

- 只能访问工作区内资源
- 命令执行要有白名单或策略控制
- 所有工具调用都要记录
- 危险操作需要明确授权或二次确认

---

### 4.6 Workspace / Sandbox

#### 职责

- 保存项目文件
- 运行构建和测试
- 执行命令
- 提供预览环境

#### 推荐形态

MVP 阶段可以先用：

- 本地工作目录
- 受控 shell
- 简单预览进程

后续再升级为：

- Docker 沙箱
- 独立执行容器
- 隔离的预览服务

---

### 4.7 State Store

#### 保存内容

- 用户会话
- 项目元数据
- 文件索引
- 任务计划
- 变更历史
- 版本快照信息
- 日志索引

#### 实现方式

按需选择：

- 简单 JSON 文件
- 本地持久化
- SQLite
- PostgreSQL
- Redis

MVP 不要求一上来就上完整数据库体系。

## 5. 任务流设计

Agent 软件的关键不是接口，而是任务流。

### 5.1 标准任务流

1. 用户提交需求
2. Agent 理解目标
3. Agent 生成计划
4. Context Builder 收集相关信息
5. Model Router 调用模型
6. Tool Gateway 执行文件或命令操作
7. Workspace 返回结果
8. Agent 检查结果
9. Agent 汇总输出给用户
10. 必要时进入下一轮修改

### 5.2 任务状态

建议任务状态至少包括：

- `pending`
- `planning`
- `running`
- `verifying`
- `done`
- `failed`
- `needs_confirmation`

## 6. 通信方式

### 6.1 对外通信

Web 层和系统之间可以使用：

- HTTP API
- WebSocket
- SSE

其中：
- HTTP API 适合初始化、查询、保存类操作
- WebSocket / SSE 适合任务流、日志流、模型输出流

### 6.2 内部通信

内部不必强制 HTTP API，可以采用：

- 直接函数调用
- 模块间消息传递
- 事件总线
- 本地任务调度
- CLI 执行结果回传

### 6.3 设计原则

- API 可以存在，但不是唯一中心
- 任务执行过程优先于接口形式
- 能直接用模块调用的，不必绕成 API

---

## 7. 安全与边界

### 7.1 文件边界

- 所有文件读写都必须限制在工作区内
- 不能访问工作区外部敏感路径

### 7.2 命令边界

- 命令执行必须受控
- 高风险命令要限制
- 超时时间必须可配置
- 输出要截断和结构化

### 7.3 模型边界

- 模型不能直接访问底层系统
- 模型只能通过工具层间接操作
- 模型的建议不等于最终执行结果，执行前后都要验证

### 7.4 回滚能力

- 每次关键修改前都应保存快照
- 出错时可按文件或按任务回滚
- 要保证 Agent 做坏事后可恢复

---

## 8. MVP 落地建议

### 8.1 第一阶段

先做成一个轻量单体：

- Web UI
- Agent Orchestrator
- Tool Gateway
- Workspace 管理
- 简单状态保存
- 基础日志

### 8.2 第二阶段

如果规模变大，再拆出：

- 独立执行服务
- 独立模型路由服务
- 独立状态存储服务
- 独立预览服务

### 8.3 推荐路线

- 先单体
- 再模块化
- 最后按需要拆服务

不要一开始就微服务化。

## 11. 一句话总结

这个架构的本质是：**Web 负责展示，Agent 负责思考，工具层负责执行，工作区负责承载结果。**
