# 上下文工程优化项目总结

## 背景

本次优化依据迭代三设计文档中“上下文工程优化”“跨会话知识共享”“分层记忆机制”的方向，对 `aicoding_ts` 的上下文构建链路做了增强。目标是在不大幅改动现有 Agent 架构的前提下，提高文件召回质量、降低无关上下文占用，并让项目记忆能够按当前任务注入到系统提示词中。

## 已完成优化

### 1. 文件相关性评分 V2

修改文件：`packages/context-builder/index.ts`

原实现主要依赖文件路径关键词匹配。本次扩展为多因素评分：

- 路径 token 与 prompt、选中文件 token 的匹配。
- 文件内容 token 命中，按命中数量加权并设置上限，避免长文件天然占优。
- 选中文件强加权，确保用户正在编辑或查看的文件优先进入上下文。
- 配置文件、源码文件、测试文件、Markdown 文档按任务关键词动态加权。
- 为每个入选文件记录简短 reason，输出到上下文块头，便于排查为什么某文件被召回。

### 2. 动态上下文预算

修改文件：`packages/context-builder/index.ts`

新增任务复杂度判断：

- `simple`：压缩预算，减少无关文件。
- `medium`：使用默认预算。
- `complex`：扩大文件数与字符预算，用于重构、架构、上下文、记忆等复杂任务。

该策略替代单一固定 `maxFiles=8`、`maxChars=12000` 的做法，使上下文规模与任务复杂度匹配。

### 3. 长文件相关片段切片

修改文件：`packages/context-builder/index.ts`

原实现只保留文件头尾，容易漏掉中间的关键函数或配置。本次改为：

- 对每一行按 prompt token 命中数评分。
- 选取高分行周围窗口。
- 多个片段之间用 `... (relevant section gap) ...` 标识省略区域。
- 如果文件没有明显命中，再回退到头尾摘要。

这样可以在相同 token 预算内保留更高价值的代码片段。

### 4. 项目记忆按任务检索

修改文件：

- `packages/context-builder/index.ts`
- `packages/agent-core/index.ts`

现有 `session-store` 已经提供 `readProjectMemory()`，但之前系统提示词会直接注入完整项目说明。本次新增：

- 将 `project-memory.md` 按 Markdown 标题拆成知识段。
- 根据当前 prompt token 对知识段评分。
- 优先注入相关段落，知识量较小时全量注入。
- 在 `agent-core` 中通过 `buildForPrompt(prompt, selectedFile, { projectMemory })` 传入项目记忆。
- 系统提示词使用“项目记忆（按当前任务检索）”作为独立区域。

这为后续跨会话知识共享打下了基础，也避免项目记忆膨胀后挤占工作区上下文。

### 5. 上下文去重与可观测性

修改文件：`packages/context-builder/index.ts`

新增按路径去重，修复原先对对象使用 `Set` 无法真正去重的问题。同时在 `contextBudget` 中增加 `strategy` 字段，标记本轮使用 `compact`、`balanced` 或 `deep` 策略。

## 后续建议

1. 增加 `packages/context-builder/__tests__/scoring.test.ts`，覆盖文件评分、动态预算和项目记忆检索。
2. 将 `contextBuilderInternals` 中暴露的内部函数用于单元测试，后续可改为更正式的测试导出。
3. 当项目文件数量超过 50 且关键词召回效果不足时，再引入 embedding/RAG 检索，避免过早增加部署复杂度。
4. 项目记忆可继续扩展写入入口，例如在任务总结后让用户确认是否保存某条经验到 `project-memory.md`。
